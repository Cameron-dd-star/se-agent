# SmartX 日报/周报生成 Skill

## 你的任务

根据用户提供的工作内容，判断报告类型，按照对应模版整理成规范日报或周报，输出 Markdown 文本，并归档到记录文件中。

---

## 第一步：判断报告类型

根据用户描述，判断属于以下哪种类型：

| 类型编号 | 类型名称 |
|---|---|
| 1 | 技术交流支持 - 初次拜访 |
| 2 | 技术交流支持 - 非初次拜访 |
| 3 | POC 测试 |
| 4 | 项目类报告 |
| 5 | 内部会议 |
| 6 | 外部会议 |
| 7 | 周报 |

如果类型不明确，先问用户确认。

> **如果是类型 7（周报）**，跳过第二步至第四步，直接进入第五步的"模版 7：周报"处理流程。

---

## 第二步：默认人员规则

在填写 SmartX 参会人员时，遵循以下规则：

- **技术人员（默认）**：薛金涛
- **销售人员（默认）**：
  - 客户为**江苏非省属医疗客户**：邢皓
  - 客户为江苏省属医疗客户：郝绪军
  - 客户属地为**安徽**：鲁成凯
  - 客户为BOE/京东方相关：庞佳泓
  - 其他地区：向用户询问

若用户已明确提供了参会人员，以用户提供的为准，不套用默认值。

---

## 第三步：缺失信息的处理规则

**核心原则**：有什么写什么，缺什么问什么，不要用占位符或空白项凑数。

- 对于用户**未提及的模块**，在初版日报中直接**省略该项**，不留空行占位
- 在日报输出完成后，用自然、亲切的语气，将缺失的关键信息**汇总提醒一次**（不要一条一条打断用户）
- 用户补充后，更新对应内容；用户表示不需要补充，则最终版本中不体现该项
- 整理输出的语言要**自然连贯**，像人写的会议纪要，而不是机器列表

---

## 第四步：初次拜访的客户背景信息

对于**类型 1（初次拜访）**，需要补充客户的公司经营业务和营收概况：

1. **优先通过网络搜索**（使用 WebSearch 工具）获取该公司的主营业务、营收规模等信息
2. 整理为约 **100 字**的简洁描述
3. 若搜索结果不足，在会议纪要的"客户概况"部分注明"（待补充）"，并提醒用户核实

### 模版 7：周报

周报的生成逻辑与日报不同，用户通常会一次性提供整周的多项工作内容。你的职责是将这些内容梳理、归类，填入以下固定结构中。

**周报格式规则：**

- 每个板块的条目用"工作项N："逐条列出，序号从 1 开始连续编号
- **【本周工作计划执行情况】** 中，每个工作项展示"工作类型-客户/事项名称"，有明确销售阶段时在末尾附加【阶段】标注
- **【本周重点工作总结】** 中，对应相同序号，每个工作项用一两句话描述本周的实际进展或关键结论
- **【下周工作计划】** 中，列出下周计划开展的工作，格式同执行情况（工作类型-客户/事项）
- **【MBO进展】** 简要说明本周 MBO 推进状态（如"正常推进中"或具体进展）
- **【POC进展】** 更新 SFDC 上未到完成阶段的 POC 本周进展；若无 POC，填写"暂无POC"

**工作类型参考（根据用户描述自动判断）：**

| 工作类型关键词 | 说明 |
|---|---|
| 外部会议 | 与客户、合作伙伴等外部人员的正式会议 |
| 技术交流支持 | 技术拜访、方案讲解、需求调研等 |
| 资料整理 | 文档、方案、材料的整理与准备 |
| POC 支持 | POC 测试过程中的技术支持 |
| 内部会议 | 团队内部的会议或协调 |
| 项目跟进 | 在建项目的进度推进与协调 |

若用户描述的工作类型不在上表中，按用户原文保留，不强行套用。

**客户等级标注规则：**

- 只有涉及具体客户商机的工作项才需要在末尾加【】标注其当前阶段（如【三类】【二类】【一类】【待定】）
- 纯内部工作（如内部会议、资料整理无对应客户）不加标注
- 若用户未提及等级，填写【待定】

**信息收集原则：**

用户提供的内容可能不完整。若以下信息缺失，在生成完初版周报后，统一询问一次：
- 某工作项的客户阶段（仅涉及商机时）
- MBO 具体进展（若用户未提及）
- POC 进展（若用户提到有 POC 但未给进展描述）

**模版如下：**

```
【本周工作计划执行情况】：
工作项1：[工作类型]-[客户名称/事项描述]【[阶段]】
工作项2：[工作类型]-[客户名称/事项描述]
...

【本周重点工作总结】：
工作项1：[本工作项的关键进展、结论或成果，一两句话]
工作项2：[本工作项的关键进展、结论或成果，一两句话]
...

【下周工作计划】：
工作项1：[工作类型]-[客户名称/事项描述]
工作项2：[工作类型]-[客户名称/事项描述]
...

【MBO进展】：[状态描述，如"正常推进中"或具体进展内容]

【POC进展】（在SFDC上未到完成阶段的POC均需要更新本周进展）：[各POC进展，或"暂无POC"]
```

> **注意**：
> - 【本周工作计划执行情况】与【本周重点工作总结】中的工作项序号一一对应，不能错位
> - 执行情况只写"做了什么"（工作类型 + 客户），总结才写"结果/进展如何"
> - 若用户提供了原始的执行情况列表，执行情况部分按用户原文整理（保留其措辞风格），总结部分根据用户描述的内容提炼

**周报输出顺序：**

1. 直接输出完整周报 Markdown 文本（不需要归档到客户记录文件）
2. 如有缺失的关键信息，在周报后用一段自然的语气汇总提问，不要拆分多次打断

> 周报无需执行常规日报的第六步（客户拜访归档）和第七步（医院客户专项归档）。

## 第六步：归档处理

日报生成完成后，进行归档：

### 文件结构

所有归档文件保存在用户工作目录（或当前输出目录）下：

```
客户拜访记录/
├── 客户拜访记录索引.md          ← 统一索引文档
├── [客户A名称]客户记录.md       ← 按客户分类的详细记录
├── [客户B名称]客户记录.md
└── ...
```

### 客户详细记录文件（`[客户名称]客户记录.md`）

每次生成日报后，**凡涉及具体客户的工作内容**（包括但不限于：拜访、技术交流、POC 测试、项目跟进、外部会议、以及内部会议中明确涉及某客户议题的部分），均将完整日报内容**追加写入**对应客户的记录文件。若文件不存在则新建，格式如下：

```markdown
# [客户名称] 客户记录

---

## [日期] [报告类型标题]

[日报完整内容]

---

## [日期] [报告类型标题]

[日报完整内容]
```

### 索引文档（`客户拜访记录索引.md`）

每次更新后，同步维护统一索引文档。**索引内容按以下顺序排列：**

1. 两周工作频次统计（最前）
2. 客户列表（居中）
3. 工作详细记录（最后）

格式如下：

```markdown
# SmartX 客户工作记录索引

> 最后更新：[日期]

---

## 两周工作频次统计

> 统计周期：以两周为一个单位，按自然周计算。每个周期单独一张表，最新周期在最上方。
> 统计范围：所有与客户相关的工作，包括拜访、技术交流、POC 测试、内部会议（涉及该客户议题）、外部会议等。

### [最新日期区间，如 2026-04-13 ~ 2026-04-26]

| 客户名称 | 属地 | 销售 | 工作次数 |
|---|---|---|---|
| [客户A] | 江苏 | 邢皓 | 2 |
| [客户B] | 安徽 | 鲁成凯 | 1 |

### [上一个日期区间，如 2026-03-30 ~ 2026-04-12]

| 客户名称 | 属地 | 销售 | 工作次数 |
|---|---|---|---|
| [客户A] | 江苏 | 邢皓 | 1 |
| [客户C] | 安徽 | 鲁成凯 | 2 |

---

## 客户列表

### [客户A名称]
- 属地：[省份]
- 销售负责人：[姓名]
- 工作记录：[N] 次
- 最近工作：[日期] - [报告类型]
- 详细记录：→ [[客户A名称]客户记录.md]

### [客户B名称]
...

## 工作详细记录

> 全量记录所有工作内容（含拜访、会议、POC、项目等），按时间戳降序排列，以双周为单位分隔。

### [最新双周区间，如 2026-04-13 ~ 2026-04-26]

- [2026-04-18] [客户A] 技术交流支持 → [[客户A名称]客户记录.md]
- [2026-04-16] 内部会议：[会议主题]（涉及客户：[客户B]）→ [[客户B名称]客户记录.md]

### [上一双周区间，如 2026-03-30 ~ 2026-04-12]

- [2026-04-10] [客户D] 初次拜访 → [[客户D名称]客户记录.md]
...
```

**统计表维护规则：**

- 每个双周周期单独一张表，**最新周期排在最上方**
- 统计范围：所有涉及具体客户的工作，包括拜访、技术交流、POC、外部会议、内部会议中明确涉及某客户议题的情况。与客户无关的纯内部会议不计入
- 每张表只列出该周期内**有工作记录（次数 ≥ 1）的客户**，次数为 0 的客户不出现在当期表中
- 新增一条记录时，找到当前所属的双周周期，更新（或新建）对应的分表
- 旧周期的表格**不修改**，保持历史记录不变

**工作详细记录维护规则：**

- 每新增一条日报，在"工作详细记录"对应的**双周区间**分节下，按**时间戳降序**插入一行链接条目
- 条目格式：`- [YYYY-MM-DD] [客户名称或"内部"] [报告类型] → [对应记录文件链接或说明]`
- 内部会议若涉及某客户，括号内注明客户名称，链接指向该客户记录文件
- 内部会议若无关联客户，末尾注明"（无关联客户）"，不生成独立客户记录
- 不设单独的内部会议模块，所有类型的工作条目统一汇聚在此处，按时间降序排列，双周分隔

## 第七步：医院客户专项归档（如适用）

在完成第六步的常规归档后，判断本次日报的客户**是否为医疗机构（医院、卫生院、诊所、医疗中心等）**。

**如果是医院类客户**，执行以下额外归档：

### 1. 查找医院所在省市

使用 **WebSearch** 搜索该医院的所在省份和地市，例如搜索 `"xx医院 所在城市 地址"`，获取准确的**省份名称**和**地市名称**（精确到地级市，如"南京市"、"合肥市"）。

### 2. 定位或新建省级医疗记录文件

归档文件路径：`客户拜访记录/[省份名]医疗客户记录.md`

例如：`江苏省医疗客户记录.md`、`安徽省医疗客户记录.md`

- 若文件**不存在**，新建该文件，顶部标题为 `# [省份名]医疗客户记录`
- 若文件**已存在**，读取现有内容，在适当位置追加，不重建已有标题

### 3. 按层级写入归档内容

```markdown
# [省份名]医疗客户记录

## [地市名]

### [医院名称]

#### [日期] [报告类型]

[日报完整内容]

---
```

**追加规则**（防止重复建标题）：
- 若文件中**已存在对应地市的 `##` 标题**，不重复创建，直接在该标题下找到或新建医院三级标题
- 若文件中**已存在对应医院的 `###` 标题**，不重复创建，直接在该标题下追加本次记录
- 地市和医院名称精确匹配（统一用全称，如"南京市"而不混用"南京"）
- 每条记录末尾加 `---` 分隔线，保持可读性

### 4. 在归档确认中说明

最终输出的归档确认部分，需额外注明：已将该记录同步写入 `[省份名]医疗客户记录.md`，位于 `## [地市名] > ### [医院名称]` 下。

---

## 输出顺序

1. 先输出**完整的日报 Markdown 文本**（直接可复制使用）
2. 再输出**归档确认**（说明写入了哪个文件；若为医院客户，额外说明已同步写入医疗客户记录文件及具体位置）
3. 最后，如有缺失的关键信息，**用一段自然的提问**提醒用户补充，汇总在一起问，不要拆分多次打断

---

## 语言风格要求

- 会议纪要部分要像有经验的销售/技术人员手写的，**自然流畅，有上下文逻辑**
